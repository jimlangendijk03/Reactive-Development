# Reactive Development: How Security Flaws Influence Update Speed

Reproducibility code for the analysis of release intervals following vulnerable and non-vulnerable versions in **PyPI**, **npm**, and **Maven**. The pipeline combines an Open Source Vulnerabilities (OSV) database snapshot with release timestamps retrieved from the three package registries, builds version-level caches, computes descriptive statistics and non-parametric tests, and displays the figures used in the analysis.

## Repository structure

```text
.
├── main.py                         # Complete data-processing and analysis pipeline
├── pyproject.toml                  # Python metadata and explicit runtime dependencies
├── uv.lock                         # Generated by uv after dependency resolution
├── .python-version                 # Recommended Python version
└── OSV data/                       # Link to download the current OSV data set
```

## Requirements

- A 64-bit operating system with enough memory and disk space for the OSV archive and JSON caches.
- Python **3.11–3.13**. The repository pins Python 3.11 in `.python-version`; `uv` can install it automatically when needed.
- [`uv`](https://docs.astral.sh/uv/) for environment and dependency management.
- Internet access when building the release cache, because the pipeline queries the PyPI JSON API, npm registry API, and Maven Central Search API.
- The OSV full-database ZIP archive named `all.zip`.

## Python dependencies

All runtime dependencies are declared in `pyproject.toml`. Running `uv sync` creates or updates `uv.lock`; commit that generated lockfile to pin the exact dependency resolution used for the paper.

| Dependency | Purpose in this project |
|---|---|
| `requests` | HTTP requests to PyPI, npm, and Maven Central |
| `tqdm` | Progress bars during OSV loading and registry fetching |
| `orjson` | Fast parsing of OSV JSON records |
| `packaging` | Parsing and ordering package versions |
| `pandas` | Tabular summaries and post-hoc-test input |
| `matplotlib` | Histograms, time-series plots, and severity boxplots |
| `numpy` | Numerical calculations and descriptive statistics |
| `cvss` | Converting CVSS v2/v3 vectors to numeric severity scores |
| `scipy` | Wilcoxon, normal-distribution, and Kruskal–Wallis functions |
| `scikit-posthocs` | Dunn post-hoc tests with Holm correction |

Python standard-library modules such as `json`, `zipfile`, `datetime`, `threading`, and `concurrent.futures` require no separate installation.

## 1. Install uv

Follow the installation instructions for your platform in the official uv documentation, then confirm that it is available:

```bash
uv --version
```

## 2. Create the environment

From the repository root, run:

```bash
uv sync
```

This creates `.venv/`, resolves the dependencies from `pyproject.toml`, and creates `uv.lock`. Commit `uv.lock` to the repository. After the lockfile has been committed, collaborators and CI should use:

```bash
uv sync --locked
```

When intentionally updating dependencies, run `uv lock --upgrade`, test the pipeline, and commit the resulting lockfile change.

## 3. Prepare the data

Download the OSV full database archive and place it at:

```text
data/osv/all.zip
```

The default paths in `main.py` are repository-relative:

```python
OSV_PATH = "data/osv/all.zip"
RELEASE_CACHE_PATH = "data/cache/release_cache.json"
VULN_CACHE_PATH = "data/cache/vulnerability_cache.json"
```

The actual values are constructed relative to the location of `main.py`, so the command may be launched from any working directory. Change these constants only when the files are stored elsewhere.

> **Exact-paper reproducibility:** the paper used a fixed OSV snapshot downloaded in September 2025. Registry release metadata was retrieved in January 2026 for packages and versions in that snapshot. A newly downloaded OSV archive is a different dataset and can therefore produce different counts and results. For an exact reproduction, use the archived September 2025 OSV ZIP and, preferably, the original release cache.

## 4. Choose a run mode

The mode is controlled by two constants near the top of `main.py`:

```python
FORCE_REBUILD_RELEASE_CACHE = False
FORCE_REBUILD_FINAL_CACHE = False
```

### A. Run the analysis from an existing processed cache

Use this when `data/cache/vulnerability_cache.json` is already present:

```python
FORCE_REBUILD_RELEASE_CACHE = False
FORCE_REBUILD_FINAL_CACHE = False
```

Run:

```bash
uv run python main.py
```

### B. Rebuild only the processed vulnerability cache

Use this after changing OSV processing logic while reusing the registry metadata cache:

```python
FORCE_REBUILD_RELEASE_CACHE = False
FORCE_REBUILD_FINAL_CACHE = True
```

Then run:

```bash
uv run python main.py
```

If `release_cache.json` does not exist, the code creates it by querying the registries.

### C. Rebuild both caches

Use this for a complete data-collection run:

```python
FORCE_REBUILD_RELEASE_CACHE = True
FORCE_REBUILD_FINAL_CACHE = True
```

For a genuinely clean rebuild, first remove any existing release cache:

```bash
rm -f data/cache/release_cache.json data/cache/vulnerability_cache.json
uv run python main.py
```

PowerShell equivalent:

```powershell
Remove-Item data/cache/release_cache.json, data/cache/vulnerability_cache.json -ErrorAction SilentlyContinue
uv run python main.py
```

A full run can take a long time because it loads the complete OSV archive and performs registry requests for every selected package. The APIs may throttle requests; the script retries common transient HTTP errors with exponential backoff and stores fetched results in the release cache.

After cache construction finishes, set both rebuild flags back to `False` and run the script again to execute the analyses.

## 5. Analysis output

Analysis mode prints the following to standard output:

- dataset composition by ecosystem;
- vulnerability severity distributions;
- npm CVE coverage by severity;
- release-interval descriptive statistics;
- severity-level update-speed tables;
- Wilcoxon signed-rank tests;
- Jonckheere–Terpstra trend tests;
- Kruskal–Wallis tests and, when significant, Dunn tests with Holm correction.

The script also opens Matplotlib windows for the quarterly update-speed figure, severity boxplots, and update-interval histograms. Close a figure window to allow execution to continue. The current code displays figures but does not automatically save image files.

For a non-interactive server, the numerical analyses can be run with a non-GUI Matplotlib backend:

```bash
MPLBACKEND=Agg uv run python main.py
```

This suppresses interactive windows; because the code calls `plt.show()` rather than `plt.savefig()`, it does not create figure files in this mode.

## Reproducibility checklist

Before reporting reproduced results, record:

1. the Git commit hash;
2. the OSV snapshot date and archive checksum;
3. the `uv.lock` version committed with the run;
4. the Python version shown by `uv run python --version`;
5. whether the release cache was reused or rebuilt;
6. the values of `START_YEAR` and the analysis end year (2025 in the supplied code).

## Troubleshooting

**`vulnerability_cache.json` is missing**  
Set `FORCE_REBUILD_FINAL_CACHE = True`, ensure `data/osv/all.zip` exists, and rerun the pipeline.

**Registry requests fail or return incomplete data**  
Retry later and retain `release_cache.json`; completed package lookups are reused on the next run. Corporate proxies or firewalls must allow access to PyPI, npm, and Maven Central.

**A cache path cannot be written**  
The adapted code creates the parent cache directory automatically. Verify that the repository directory is writable and that sufficient disk space is available.

**The program appears to pause during analysis**  
A Matplotlib figure window is probably open. Close it to continue to the next plot or test.
